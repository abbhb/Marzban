# 用户门户、钱包与套餐设计

本文定义 Marzban MGMA fork 的自助用户门户。实现必须遵循这里的数据、权限和事务契约，不能把门户用户当成面板管理员，也不能把钱包余额当作可直接覆盖的普通字段。

## 1. 范围与角色

- 门户用户可以注册、登录，查看余额、钱包流水、当前套餐、可购买套餐、本人流量，并获取或撤销本人的 MGMA 临时订阅链接。
- sudo 管理员可以维护套餐及全局可见性、充值、覆盖开通任一套餐和续期。
- 普通 Marzban 管理员不获得钱包或门户管理权限。
- 本版本不接入外部支付网关；充值只能由 sudo 管理员完成。

## 2. 核心规则

1. 一个门户账户最多关联一个 Marzban 代理用户，并最多只有一个当前生效订阅。购买或由管理员开通其他套餐时直接覆盖该订阅。
2. 注册只创建门户账户。首次开通套餐时才创建同名 VLESS 用户，避免未付费账户进入 Xray 配置。
3. 套餐可设置是否对门户用户可见。所有注册用户默认看到全部全局可见套餐；隐藏套餐不出现在自助列表且不能自助购买，但 sudo 管理员仍可为指定用户开通。
4. 自助购买按“覆盖”处理：余额扣除成功后，以购买时刻重新计算到期时间，流量归零，覆盖流量上限与 inbound 权限；已有 VLESS UUID 保持不变。
5. 管理员续期不扣钱包，只把当前到期时间向后延长指定天数；已过期套餐从当前时间开始延长。
6. 金额以最小货币单位存储，当前货币固定为 `CNY`，例如 `1299` 表示 `¥12.99`。禁止用浮点数保存金额。
7. 每次余额变化必须写入钱包流水。充值金额为正数，购买金额为负数；流水保存变更后余额和操作人，不能编辑或删除。
8. 自助购买、管理员充值、覆盖开通和续期都要求 `Idempotency-Key`。同一账户重复提交同一个键只返回第一次结果，不再次扣款、充值或延长；复用键但更换关键参数会返回冲突。

## 3. 数据模型

### `portal_accounts`

- 登录名、密码摘要、启用状态；
- 当前钱包余额（分）；
- 一对一代理用户 `user_id`；
- `password_reset_at` 用于使旧 token 失效。

### `subscription_plans`

- 名称、说明、价格、货币、有效天数、流量上限；
- VLESS inbound tag 列表；
- 全局可见状态 `is_visible`；
- 已有购买记录引用的套餐只隐藏，不硬删除。

### `portal_subscriptions`

- 每个账户至多一条当前套餐；
- 当前展示名称引用关联套餐，套餐改名后门户与管理端同步显示新名称；
- 价格、货币、有效天数、流量上限和 inbound tag 保存购买或开通时快照；
- 保存购买时间、当前开始/到期时间和最近一次购买；
- 自助购买与管理员覆盖开通都会覆盖该记录，管理员续期只更新到期时间。

### `portal_purchases`

- 不可变购买历史；
- 保存幂等键、价格、余额前后值和交易发生时的套餐名称及权益快照。

### `wallet_transactions`

- 不可变余额流水；
- 保存有符号金额、变更后余额、类型、管理员操作人、购买引用、备注和时间。

## 4. API 边界

门户接口使用独立的 `portal` JWT，不能调用 `/api/admin/*`。除注册和登录外，所有门户接口都从 token 获取当前账户，不接收可替换的用户名。

- `POST /api/portal/register`
- `POST /api/portal/token`
- `GET /api/portal/me`
- `GET /api/portal/plans`
- `GET /api/portal/wallet/transactions`
- `POST /api/portal/purchase`
- `POST /api/portal/mgma`、`DELETE /api/portal/mgma`

sudo 管理接口统一位于 `/api/commerce/admin/*`：

- 套餐列表、新建和修改；
- 门户账户列表；
- 充值；
- 显式选择任一套餐进行覆盖开通、续期和停用当前套餐。

## 5. 事务与安全

- 购买用数据库条件更新完成原子扣款：只有 `balance >= price` 时余额才会变化。
- 扣款、钱包流水、购买历史、当前套餐和代理用户覆盖在同一数据库事务内提交；任一步失败均回滚。
- Xray 更新只在数据库提交后执行。失败时数据库仍是事实来源，管理员可以重试同步，不重复扣款。
- 门户密码使用与管理员相同的 bcrypt 实现，注册密码至少 10 个字符；响应和日志不得包含密码摘要、JWT、MGMA token 或 VLESS UUID。
- 注册按真实客户端 IP 限制为每分钟 5 次；登录按 IP 限制为每 5 分钟 60 次，并按 IP + 用户名限制为每 5 分钟 5 次，登录成功后清除该账号失败窗口。限流器最多保留 10000 个键，防止攻击者用随机用户名耗尽内存。
- MGMA 明文 token 仍只在签发响应中出现一次，数据库只保存 keyed digest；门户只可为自己的代理用户签发。

## 6. 邀请制注册与 IP 防护

- 公开注册永久关闭；`POST /api/portal/register` 必须提交管理员生成的邀请码。
- 邀请码支持立即或指定时间生效、指定时间过期或永久有效，并可限制为一次、N 次或不限次数。明文使用 32 字节随机熵，只在创建响应展示一次；SQLite 只保存 SHA-256 摘要和短前缀。
- 邀请次数通过条件 `UPDATE` 在注册事务内原子递增。账户创建或使用审计失败会回滚次数；同一邀请码并发注册不能越过 `max_uses`。
- 系统会先验证邀请码，再检查用户名冲突；无有效邀请码时统一返回不可注册，不泄露门户或代理用户名是否存在。
- `portal_invitation_uses` 记录邀请码、已创建账户、真实来源 IP 和使用时间，但不记录邀请码明文。
- 门户登录、管理员登录、无效/畸形注册按真实来源 IP 记录持久化失败次数。默认策略为登录 15 分钟 8 次、注册 10 分钟 5 次，达到阈值自动封 24 小时；管理员可修改阈值、统计窗口和封禁时长，`0` 表示永久封禁。
- 自动封禁与手工封禁统一写入 `portal_ip_blocks`，保存精确 IP/CIDR、来源、创建人、到期时间及不可为空的原因。管理后台可以提前解除封禁；精确 IP 解封同时清除对应失败计数，避免立即再次触发。
- 黑名单在业务路由解析请求体之前执行，覆盖所有 `/api/portal/*`、管理员登录和公开订阅路径，因此畸形 JSON/Form 也不能绕过。sudo 黑名单管理 API 不受该中间件拦截，以便持有有效管理员会话时处理误封；无法登录时仍可通过服务器 SSH 恢复。
- 客户端 IP 沿用 MGMA 的可信边界：Unix socket 部署只信任 Nginx 覆盖写入的 `X-Real-IP`，TCP 直连只认真实 peer，不信任客户端伪造的转发链。无法解析真实 IP 时认证/注册请求失败关闭。
- 攻击者可控的失败计数表最多保留 10000 行；超过上限先删除过期窗口，再淘汰最旧项，避免随机 IP 撑爆 SQLite。黑名单命中后不为每次拒绝写数据库，避免形成写放大。
- 可选宿主机服务每 15 秒把有效黑名单同步到无 timeout 的 nftables IPv4/IPv6 interval set，从网络层丢弃来源到面板 TCP 端口的连接。同步失败保留上一版策略，完整卸载才删除专用表；详见 `docs/HOST-IP-GUARD.zh-CN.md`。

sudo 管理接口：

- `GET/POST /api/commerce/admin/invitations`
- `POST /api/commerce/admin/invitations/{id}/disable`
- `GET/POST /api/commerce/admin/security/blocks`
- `POST /api/commerce/admin/security/blocks/{id}/revoke`
- `GET/PUT /api/commerce/admin/security/settings`

## 7. UI

- `#/portal/register`、`#/portal/login` 和 `#/portal` 构成用户门户；注册页明确提示公开注册关闭并要求邀请码。
- 门户首页展示余额、当前套餐、到期时间、已用/总流量、全部全局可见套餐、钱包流水和 MGMA 获取按钮。
- 管理端“用户与套餐”页面提供套餐可见性维护、为账户选择套餐并覆盖开通、充值、续期、邀请码生成/停用、自动封禁策略和 IP 黑名单管理。
- 所有新增文案必须同时加入 `en/fa/ru/zh` locale，构建测试会校验 key 集合一致，避免出现原始国际化 key。

## 8. `v0.8.4-mgma.6` 迁移语义

- Alembic revision `b7f3e1a9c204` 删除账户级 `assigned_plan_id`，把旧套餐的 `is_active` 原值迁移为 `is_visible`，并删除不再需要的 `is_default`。
- 迁移不会删除套餐、门户账户、钱包余额与流水、购买历史、当前订阅或代理用户；旧版已上架套餐升级后保持可见，旧版下架套餐升级后保持隐藏。
- 旧的“默认套餐”和账户级分配关系没有对应的新语义，升级后不保留。注册账户直接读取全局可见列表，管理员开通时必须显式选择套餐。
- 回退 migration 会恢复旧字段，但无法重建已经删除的默认/账户分配关系；生产回退仍优先恢复升级前 SQLite 备份与旧镜像。
